HOW TO USE IT (Even Now)
================================================================================

Organizations can START using the framework design:

1. In your WORKSPACE:
   git_repository(
       name = "michelangelo",
       remote = "https://github.com/michelangelo-ai/michelangelo.git",
       commit = "COMMIT_SHA",
   )

2. Create extension protos:
   your-org/extensions/project_ext.proto

3. Use Bazel rule:
   load("@michelangelo//bazel/rules/proto:patched_proto.bzl", "patched_proto_library")

   patched_proto_library(
       name = "michelangelo_extended",
       base_protos = "@michelangelo//proto/api/v2:v2_proto",
       extension_protos = glob(["extensions/*.proto"]),
       field_prefix = "YOUR_ORG_",
   )

4. Import in code:
   import v2 "your-org.com/michelangelo/proto/api/v2"

   project := &v2.Project{
       Spec: &v2.ProjectSpec{
           YOUR_ORG_owner_id: "uuid",
       },
   }

ARCHITECTURE SUMMARY
================================================================================

Two Repository Model:

  OSS Michelangelo (GitHub)          Uber Internal (go-code)
  ├─ Base protos                     ├─ Extension protos only
  ├─ Patch compiler                  ├─ BUILD using OSS rule
  ├─ Bazel rules                     └─ Services using patched protos
  ├─ Code generators
  └─ Documentation

Build Flow:
  1. Fetch OSS Michelangelo (@michelangelo)
  2. Run config-generator (auto-config)
  3. Run proto-patcher (merge base + extensions)
  4. Create proto_library (patched protos)
  5. Run protoc-gen-kubeproto (CRD code)
  6. Run protoc-gen-validation (validation code)
  7. Run protoc-gen-yarpc (RPC code)
  8. Compile to go_library
  9. Link into services

================================================================================
BENEFITS OF THIS DESIGN
================================================================================

✓ Clean Separation - OSS unchanged, extensions separate
✓ No Forking - Organizations extend without forking
✓ Easy Updates - Update OSS commit SHA, rebuild
✓ Type Safety - Generated Go code with all fields
✓ Validation - Extension fields validated like base fields
✓ CRD Integration - Extension fields in K8s schemas
✓ Multi-Org - Different orgs extend independently
✓ Documented - Comprehensive guides and examples
